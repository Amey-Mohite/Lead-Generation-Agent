01

Architecture

Two agents, composed rather than merged. A Research Agent runs an autonomous tool-calling loop to build a factual brief on a company. A Lead Orchestrator takes that brief, scores it against an Ideal Customer Profile, and — only if it clears the bar — drafts a personalized outreach email. A Discovery layer sits in front of both, turning "credit unions in the UK" into fifty real domains to run through the same pipeline. Everything downstream (persistence, the API, n8n) treats the orchestrator as the one thing that decides whether a lead is worth anyone's time.

Composed, not monolithic

The Research Agent doesn't know what an ICP is. The Orchestrator doesn't know how to search the web. Discovery doesn't qualify anything — it just proposes candidates and hands them to the Orchestrator unmodified. Each piece is replaceable on its own.

Swap the model, not the code

OpenRouter, NVIDIA NIM, OpenAI, and Anthropic sit behind one LLMProvider interface with automatic fallback. Changing providers is a one-line config edit, not a rewrite — verified against each SDK's real signatures, not assumed from memory.

The pipeline, end to end

Every lead moves through the same seven stages — cost-saving short-circuits and a human checkpoint included, not bolted on after the fact.

01 · discover
Query → candidates
One broad query becomes a batch of real candidate companies via a single structured extraction call.
02 · research
Autonomous tool loop
A ReAct-style agent searches and fetches pages to build a factual brief: summary, key facts, contacts, sources.
03 · qualify
Score vs. ICP
0–100 fit score plus reasoning, against a config-driven Ideal Customer Profile. Below threshold stops here — no draft call fires.
04 · draft
Personalized outreach
Subject + body referencing a real fact from research and a real capability of the sender — never generic copy.
05 · approve
A human decides
n8n posts the draft to Slack. Nothing sends until a person clicks approve.
06 · send
Outreach goes out
Only on approval — via Gmail, orchestrated by n8n, not the app itself.

05

Observability

Three pillars, one entry point — setup_observability() — called identically from the API and every CLI script, so behavior never depends on how the process was started.

Tracing

One Langfuse trace per lead run, nested spans for research/qualify/draft, across all four LLM providers — via a drop-in client swap for OpenAI-compatible providers and global instrumentation for Anthropic.

Metrics

Prometheus /metrics on a dedicated registry: request count and latency by route template (never a raw ID, to avoid label-cardinality blowups), plus job outcomes by kind and status.

Logging

Structured JSON on every log line, with a per-request correlation ID propagated via contextvars — no ID threaded manually through the call stack.

06

n8n orchestration

n8n never re-implements the agent — it calls the API and owns the parts that need a human or an external inbox: triggering, approval routing, and sending.

01 · Ingestion

A webhook accepts a target or query and routes it to /v1/leads or /v1/discovery — lets a non-technical trigger (a form, a Sheet row, a CRM event) start a run.

02 · Approval + send

Polls for pending drafts, posts them to Slack with approve/reject buttons, and on approval fetches the full lead and sends it via Gmail — closing the loop back to the API.

03 · Alerting

Push, not poll: the app calls an n8n webhook the instant a background job fails, which posts straight to an ops Slack channel — no counter-diffing, no detection delay.

How it was built

Eleven phases, each shipped with its own design spec, implementation plan, and code review before merge — test-driven throughout, with real bugs found and fixed by actually running the thing, not just by writing code that compiles.

1
Foundations
Config, FastAPI app factory, Postgres, health/readiness probes, Docker.
2
Multi-provider LLM layer
One provider interface across OpenRouter, NVIDIA, OpenAI, and Anthropic, with automatic fallback.
3
Research sub-agent
An autonomous tool-calling loop — web search and page-fetch tools, bounded turns, self-correcting structured output.
4
Lead Orchestrator
Qualification against a configurable ICP; drafting only fires when a lead actually clears the bar.
5
Discovery layer
One broad query fans out into many researched, qualified, drafted leads automatically.
6
Exporters
A pluggable export protocol, Excel first — the first durable, shareable artifact out of the pipeline.
7
Persistence
Postgres via Alembic migrations; permanent domain dedup so nothing gets re-researched twice.
8
API layer
Background-job-plus-polling FastAPI service, so long sweeps never hold a connection open.
9
Observability
Langfuse tracing, Prometheus metrics, structured logging — through one shared entry point.
10
n8n integration
A human-approval gate, an ingestion trigger, and push-based failure alerting.
11
Deploy
docker-compose brings up the whole stack; Kubernetes and a managed Postgres are the documented next step.

Not just "it ran once." A handful of real issues surfaced by actually exercising the system, not by inspection:

  • A connection-pool leak where a new database engine was created on every request — caught and fixed before it became a production incident.
  • An unbounded-cardinality risk in Prometheus labels (real request paths instead of route templates) — caught in review, fixed before merge.
  • A dedup bug where Discovery silently returned zero new candidates once every result had already been seen — fixed by teaching the search itself which domains to skip.
08

Deployment & stack

docker compose up brings up the app, Postgres, and n8n together. Kubernetes and a managed Postgres provider are the documented path for a real production deployment — not built here, deliberately, since this stage is about proving the system, not standing up a cluster.
